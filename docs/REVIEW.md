### Architectural observations (non-findings)

- **Dependency direction is strictly downward with no cycles.** The chain `logging → guards → errors → problem-details` flows one way (e.g. `guards.ts:5` → `errors`, `structured.ts:24`/`console.ts:21` → `guards`); `problem-details` imports nothing internal. tsup code-splitting (`tsup.config.ts:17-23`) is deliberately relied on to preserve single class identity (`NullError`, `HTTPException`, `jose` errors) across entries — a well-reasoned `instanceof`-safety decision.
- **`auth-internal` does not leak.** It imports only `jose` + its own types and is consumed solely by `hono/cloudflare-access.ts`, `vite/plugin.ts`, and `testing/index.ts` (all intended). It has no barrel and no subpath export, so it is genuinely private. The single JWT/JWKS/policy unit shared by both the Worker middleware and the Vite dev emulator is the correct DRY seam for a security-critical primitive.
- **Public API encapsulation is careful and correct.** `package.json#exports` declares exactly 8 subpaths with no wildcard; internals (`levels`, `safe-json`, `optional-field`, `internal/console`, `auth-internal`) are unexported. The root barrel (`src/index.ts:1-6`) deliberately excludes `hono`/`vite`/`testing` (peer/Node-only deps) and is guarded by a test. `testing/index.ts` re-exports only the token/cookie _builders_ a test author needs, not the verification internals — deliberate minimal surface.
- **CLI ports-and-adapters is exemplary.** `run.ts` is pure orchestration depending only on the `FileSystem`/`WranglerRunner` interfaces (`types.ts`); `fs.ts` and `wrangler.ts` are the side-effecting adapters; `index.ts:24-27` is the composition root wiring real adapters. Fully testable without I/O — textbook hexagonal separation.
- **`problem-details` factory/registry patterns are clean.** `problemDetails()` (factory) and `createProblemTypeRegistry()` (registry) both funnel through the single `ProblemDetailsError`/`normalizeProblemDetails` path; no logic duplication, no leakage of Hono types into the Hono-free core (verified against `problem-details/types.ts:2-4`).
- **Configuration handling is appropriately localized.** Env reads (`CLOUDFLARE_TEAM_DOMAIN`, `ENVIRONMENT`) happen only at the middleware boundary via small per-middleware typed interfaces (`cloudflare-access.ts:47`, `logger-middleware.ts:21`), read at request time. A library correctly declines to impose a global config validator on consumers — no finding.
- **Fail-closed security default** (`cloudflare-access.ts:218,228`) and prototype-pollution stripping (`utils.ts:9,17`) are noted as sound; detailed security review belongs to the security agent.
