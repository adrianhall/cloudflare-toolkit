### Why some auth findings survive Cloudflare Access

Cloudflare's own documentation is explicit that placing a Worker behind Access does **not** relieve the origin of validating the JWT itself:

> "Unless your application is connected to Access through Cloudflare Tunnel, your application must validate the token to ensure the security of your origin. **Validation of the header alone is not sufficient** — the JWT and signature must be confirmed to avoid identity spoofing."
> — [Cloudflare Access: Application token](https://developers.cloudflare.com/cloudflare-one/access-controls/applications/http-apps/authorization-cookie/validating-json/)

Cloudflare's own reference Worker snippet for [one-click Access for Workers](https://developers.cloudflare.com/changelog/post/2025-10-03-one-click-access-for-workers/) validates **both** `issuer` and `audience` inside the Worker (`jwtVerify(token, JWKS, { issuer, audience })`) — exactly the two checks this toolkit's `verifyAccessJwt` currently skips by default (SEC-003, and SEC-001 when `audience` is omitted).

Two concrete reasons an Access-protected deployment does not fully close these gaps:

1. **Separate, often-still-open ingress paths.** Every Worker also has a [`workers.dev` subdomain](https://developers.cloudflare.com/workers/configuration/routing/workers-dev/) and, since mid-2025, [Preview URLs](https://developers.cloudflare.com/workers/versions-and-deployments/preview-urls/) — both reachable directly, bypassing whatever Access policy is attached only to a custom domain/route. (Cloudflare shipped a dedicated "put `workers.dev`/Preview URLs behind Access" toggle precisely because this was a common gap — see the [Domains tab changelog](https://developers.cloudflare.com/changelog/post/2026-05-14-domains-tab/).) A request sent directly to one of these bypass paths, carrying a forged/replayed `Cf-Access-Jwt-Assertion` header or `CF_Authorization` cookie, reaches the Worker's `verifyToken`/`verifyAccessJwt` logic with **no edge-side Access check at all** — at that point, only the code's own `aud`/`iss`/algorithm checks (or `enableDevTokens` gate) stand between the attacker and a valid session.
2. **All Access apps in one Zero Trust team share the same JWKS.** A token that is genuinely, validly signed for a _different_ Access application in the same account will pass `verifyAccessJwt`'s signature check. Only an explicit `audience` (SEC-001) or `issuer` (SEC-003) comparison distinguishes "signed by our team" from "issued for _this specific app_" — the classic confused-deputy pattern the RS256/JWKS architecture does not prevent by itself.

For the same reason, `enableDevTokens`/`DEFAULT_DEV_SECRET` (SEC-006) and the JWKS URL-construction bug (SEC-009) live in that same self-verification code path and are not helped by an edge-level Access policy either — a misconfigured `enableDevTokens: true` reachable via any bypass path lets an attacker mint a valid session with the published HMAC secret, and the JWKS-fetch normalization bug is entirely internal to the origin's own outbound request, independent of how the inbound request was authenticated.

## Notes

- No `critical`-severity findings across any review pass.
- The two `high`-severity items (**SEC-001** optional-audience bypass, **SEC-002** CLI `shell:true` injection) are the release-blockers to address first. Of these, only **SEC-001** survives deployment behind Cloudflare Access — **SEC-002** never runs in a deployed Worker at all.
- **5 of 20 findings remain live risks even in a Cloudflare-Access-protected deployment** (SEC-001, SEC-003, SEC-004, SEC-006, SEC-009) — all are in `src/lib/auth-internal/jwt.ts` / `jwks.ts`, the Worker's own JWT self-verification code. This matches Cloudflare's own documented guidance that an Access-fronted origin must still independently validate the JWT, and that `workers.dev`/Preview URLs are separate ingress paths that are not automatically covered by a custom-domain Access policy.
- Both the Security and Code reviews independently confirmed the codebase's auth _default posture_ is fail-closed (`enableDevTokens: false`, `defaultAction: "block"`) — the residual auth risks above are hardening gaps/footguns that a bypass path (workers.dev, Preview URLs, or an accidental `enableDevTokens: true`) can still expose, rather than active bypasses under a correctly locked-down default configuration.
- The Architecture review found no dependency cycles, clean public-API encapsulation, and correct hexagonal separation in the CLI — these are recorded as non-findings in `REVIEW.md` and are not repeated here.
